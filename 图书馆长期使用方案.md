# 古籍探寻系统 - 图书馆长期使用与运维方案

> 版本：v1.0  
> 适用对象：图书馆馆长、系统管理员、馆员  
> 编制日期：2026 年 7 月

---

## 一、方案概述

"古籍探寻系统"是一款面向高校图书馆的古典文献阅读激励平台，旨在通过游戏化任务机制引导学生主动接触和阅读 CADAL（大学数字图书馆国际合作计划）中的古籍资源。本方案从系统部署、日常运维、学期规划、数据安全、人员培训、应急处置、扩展路径及成本控制八个维度，为图书馆提供一套完整的长期运行保障体系。系统采用纯前端架构，无需服务器运维，真正实现"零基础设施成本、可持续运行"。

---

## 二、系统架构与部署

### 2.1 技术架构

本系统是一个**纯前端 Web 应用**，技术栈如下：

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 前端框架 | React / Vue（标准 SPA） | 响应式设计，兼容 PC 与移动端 |
| 数据存储 | 浏览器 IndexedDB / localStorage | 所有数据存于用户本地浏览器 |
| 静态资源托管 | GitHub Pages | 全球 CDN 加速，免费 HTTPS |
| 版本管理 | Git + GitHub | 版本可追溯，支持多人协作维护 |

### 2.2 部署方式

系统部署在 **GitHub Pages**，具备以下优势：

- **零服务器**：无需购买或维护任何服务器，GitHub 免费提供静态网站托管；
- **自动 HTTPS**：证书自动签发与续期，无需人工干预；
- **全球 CDN**：用户访问速度快，不受地理位置影响；
- **一键回滚**：通过 Git 版本管理，任何时候可快速回退到历史版本；
- **持续部署**：代码推送后自动上线，更新周期以分钟计。

### 2.3 访问方式

学生和馆员通过浏览器访问固定 URL 即可使用系统，无需安装任何客户端软件。支持 Chrome、Edge、Firefox、Safari 等主流浏览器。

## 三、日常运维流程

### 3.1 馆员日常操作清单

| 序号 | 操作事项 | 频次 | 负责人 | 操作说明 |
|------|----------|------|--------|----------|
| 1 | 古籍书目上架 | 按需（新资源入库时） | 采编馆员 | 在管理后台添加古籍条目，录入书名、作者、版本、CADAL 链接等信息 |
| 2 | 阅读任务发布 | 每周 | 阅读推广馆员 | 根据教学进度和馆藏特色，发布主题阅读任务 |
| 3 | 学生名单管理 | 每学期初 | 系统管理员 | 批量导入或更新学生信息，按班级/专业分组 |
| 4 | 任务完成审核 | 每日 | 值班馆员 | 查看学生提交的阅读记录，确认任务完成情况 |
| 5 | 排行榜更新 | 每周 | 系统自动 | 系统自动计算积分排名，无需人工干预 |
| 6 | 数据备份 | 每周 | 系统管理员 | 导出系统数据并保存至图书馆内网存储 |
| 7 | 系统功能检查 | 每周 | 系统管理员 | 验证各页面加载正常、链接有效、数据读写正常 |

### 3.2 管理员后台功能一览

- **古籍管理**：添加、编辑、下架古籍条目
- **任务管理**：创建阅读任务、设置积分规则、指定参与对象
- **学生管理**：导入/导出学生名单、分组管理、账号状态管理
- **数据看板**：阅读量统计、任务完成率、积分排行、活跃度趋势
- **系统设置**：学期配置、积分规则、公告管理

以标准学期（18 周）为例，以下为各关键节点的操作安排：

| 时间节点 | 阶段 | 重点工作 | 负责角色 | 产出物 |
|----------|------|----------|----------|--------|
| 开学前 2 周 | 准备期 | 1. 更新系统版本（如有）<br>2. 导入新学期学生名单<br>3. 清空上学期积分，初始化新赛季<br>4. 制定本学期阅读推广计划 | 系统管理员 + 阅读推广馆员 | 学生名单导入完成、赛季初始化确认 |
| 第 1 周 | 启动期 | 1. 发布新学期启动公告<br>2. 发布首批阅读任务<br>3. 组织新生入馆教育（介绍系统使用） | 阅读推广馆员 | 启动公告、首批任务上线 |
| 第 2-4 周 | 运行期 | 1. 持续发布阅读任务<br>2. 跟踪学生参与数据<br>3. 解答学生使用疑问 | 值班馆员 | 任务列表、参与度周报 |
| 第 5-8 周 | 运行期 | 1. 期中数据检查<br>2. 分析各专业/班级参与率<br>3. 对低活跃班级定向推送通知 | 阅读推广馆员 | 期中阅读数据报告 |

| 假期 | 休整期 | 1. 系统常规维护<br>2. 根据反馈优化功能<br>3. 规划下学期推广策略 | 系统管理员 + 阅读推广馆员 | 系统优化方案、下学期规划 |

## 五、数据安全保障

### 5.1 数据存储机制

本系统所有用户数据（包括学生阅读记录、积分、任务进度等）均存储在**用户本地浏览器的 IndexedDB** 中。这意味着：

- **数据物理隔离**：每位学生的数据仅存在于其个人设备浏览器中，不存在中心化数据库；
- **隐私天然保护**：图书馆不收集、不持有任何学生个人阅读数据；
- **无数据泄露风险**：不存在服务器被攻击导致数据泄露的可能；
- **合规性**：符合《个人信息保护法》关于最小化收集和本地化存储的原则。

### 5.2 数据备份策略

为保障数据不丢失，建议采取以下备份措施：

| 备份方式 | 频次 | 说明 |
|----------|------|------|
| 浏览器本地存储 | 实时 | 系统自动保存，无需操作 |
| 管理员端数据导出 | 每周 | 馆员从管理后台导出汇总统计数据（不含个人明细） |
| 版本快照（GitHub） | 每次更新 | 代码和配置自动纳入 Git 版本管理 |
| 内网存储备份 | 每月 | 将导出数据存放至图书馆 NAS 或内网服务器 |

### 5.3 访问控制

- 管理后台通过密码保护，仅授权馆员可访问；
- 学生端无需登录，通过班级码或邀请码加入活动，避免个人信息采集；
- 管理密码定期更换，建议每学期至少更换一次。

## 七、应急预案

### 7.1 常见问题与处置

| 故障场景 | 影响范围 | 处置方案 | 恢复时间 |
|----------|----------|----------|----------|
| GitHub Pages 无法访问 | 全体用户 | 1. 检查网络连接<br>2. 确认 GitHub 服务状态（status.github.com）<br>3. 必要时启用备用部署（如 Vercel / Cloudflare Pages 镜像） | 30 分钟内 |
| CADAL 链接失效 | 部分古籍 | 1. 在 CADAL 平台重新检索该古籍<br>2. 更新系统中的链接地址<br>3. 如已下架，标记为"暂不可用" | 1 个工作日内 |
| 浏览器数据丢失 | 单个学生 | 1. 引导学生检查浏览器设置（是否清除了站点数据）<br>2. 如积分丢失，由馆员根据审核记录手动恢复 | 即时 |
| 管理员密码遗忘 | 管理端 | 通过预设的恢复邮箱或联系技术维护人员重置 | 1 小时内 |
| 系统功能异常 | 部分用户 | 1. 确认是否为浏览器兼容性问题<br>2. 通过 GitHub Issues 提交 Bug 报告<br>3. 技术维护人员远程修复 | 视问题复杂度 |

### 7.2 重大故障处置流程

```
发现故障 → 评估影响范围 → 发布公告（图书馆主页/微信群）→ 启动修复 → 验证恢复 → 复盘总结
```

### 7.3 备用部署方案

建议在 GitHub Pages 之外，额外配置一个备用部署渠道（如 Vercel 或 Cloudflare Pages），当主部署出现问题时可在 5 分钟内切换，确保服务不中断。
